¿Cómo estructuramos el servidor?

Teniendo en cuenta: Express + HTML vanilla, solo web, historial persistente. 3 formas de organizar el código.

A

Todo en index.ts + carpeta public/

El servidor Express vive en el mismo index.ts (reemplaza el REPL). El frontend es una carpeta public/ con archivos estáticos separados.


📁 supabase-agent/
  ├── index.ts        # Express + lógica del agente
  ├── public/
  │   ├── index.html
  │   ├── style.css
  │   └── app.js
  └── chat-history.json  # generado en runtime

Pros

  • Estructura simple y clara
  • Mínima refactorización de index.ts

Cons

  • index.ts mezcla servidor y lógica
B ⭐ Recomendada

Separar server.ts + agent.ts + public/

El agente SDK se extrae a su propio módulo. El servidor Express queda limpio. Responsabilidades claras.


📁 supabase-agent/
  ├── server.ts      # Express, rutas, SSE
  ├── agent.ts       # lógica del agente SDK
  ├── public/
  │   ├── index.html
  │   ├── style.css
  │   └── app.js
  └── chat-history.json

Pros

  • Código limpio y mantenible
  • Fácil de extender después
  • Cada archivo tiene un propósito

Cons

  • Requiere refactorizar index.ts
C

Todo inline en index.ts (HTML como string)

El HTML se sirve como template string dentro de index.ts. Sin carpeta public, todo en un archivo.


📁 supabase-agent/
  ├── index.ts        # todo: servidor + HTML + agente
  └── chat-history.json

Pros

  • Un solo archivo

Cons

  • Muy difícil de mantener
  • HTML mezclado con TypeScript
